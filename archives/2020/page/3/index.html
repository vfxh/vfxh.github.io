字节序

所谓字节序就是超过一个字节的数据类型的变量在内存中存储时字节的顺序，也就是说像char这种数据类型的变量不用担心字节序的问题了，因为char类型只占一个字节。字节序是处理器架构的一个特性，不同的处理器架构之间可能存在差异。所以，在同一台机器上的进程通信不需要考虑字节序的问题。而在不同主机之间进行通信，就需要处理字节序的问题了。字节序分为大端字节序和小端字节序，在TCP/IP协议体系中，进程间进行网络通信默认的字节是大端字节序。

大端字节序：最高有效字节出现在低地址上，最低有效字节出现在高地址上
小端字节序：最高有效字节出现在高地址上，最低有效字节出现在低地址上

比如现在有一个十六进制的整数0x12345678，有四个字节的内存块，这个数可以以这样的形式存储：0x12, 0x34, 0x56, 0x78。其中0x12就是最高有效字节，所谓的最高有效字节就是 权值最大的字节。相反的，最低有效字节就是 权值最小的字节，所以这里的0x78就是最低有效字节。

进程间通信

进程间通信（IPC）只是一组进程间进行数据交换的行为。进程的IPC是由操作系统控制的。操作系统中的IPC是运行在同一台主机上的不同进程。通信网络中的IPC可以是同一台主机上的一组不同进程，也可以是不同主机（可能运行的操作系统也不同）之间的一组进程。
网络IPC是通过一个叫做套接字（socket）的软件接口来作为报文交换的入口点和出口点，而这个套接字的就是IP和port的组合体。IP的作用是定位实体所在的位置，port的作用是定位报文交换的进程。
应用层的进程可以控制应用层的一切，但对于运输层的控制是有限的。比如可以选择用哪种运输层协议，TCP或者UDP。以及缓冲区的大小，报文段的长度大小。

散列表

基本概念

散列表

散列表也叫哈希表，是一种能够实现将多个键映射到多个值的数据结构，其添加，删除，查找等操作都是常数时间完成。其中，键和值的映射关系是通过散列函数来完成的。理想的散列表是一个固定大小的数组，不同的键经过散列函数的计算之后能够均匀分布到不同单元的散列表项当中，但这只是在理想的状态下。

散列函数

散列函数就是将任意长度的数据映射到固定长度数据的函数。散列表使用散列函数来计算键在散列表的索引，来找到键在散列表中对应的值。一个好的散列函数能够将不同的键均匀的映射到散列表的不同单元中，提高散列表利用率。所以，散列函数的好坏取决于其解决冲突的能力。但凡事都有利弊。一个好的散列函数通常会增加计算的开销，效率有所降低。所以，适当选择不是很完善的散列函数有助于提高效率。

取余法

最简单的散列函数是取余法，以键的长度对散列表的大小取余。对于键是字符串的形式，通常会先计算字符串的长度，然后根据字符串的长度对散列表的大小取余。一般来说，散列表的大小取素数会更好。取素数的散列计算方式：hash(key) = len(key) mod hashTableSize。如果散列表的长度不是素数，那么计算公式就是：hash(key) = (len(key) mod p) mod hashTableSize，p是一个大于hashTableSize的素数。

为什么取余法的散列表长度要取素数

简而言之就是为了减少冲突，使得键在散列之后能够均匀分布在散列表中

TCmalloc

其实老夫压根就没明白，就简单记下来……..

TCmalloc（thread-cache-malloc）是Google开发的内存分配器，Go的内存分配器和TCmalloc相似，了解一下有助于理解Go的内存分配。
当程序执行的时候，在操作系统中是以进程来表示。操作系统会给一个进程分配指定大小的虚拟内存，而TCmalloc管理的就是这个虚拟内存中的堆。大多数现代操作系统中，进程内部会有很多个线程在执行不同的任务。一个进程内的全部线程共享进程的全部资源（线程与线程之间的栈是独立的，但是他们栈集合就是进程的栈），如果进程内的线程在申请内存的时候都在进程的堆中分配，那么就会有很大的竞争，效率不高。TCmalloc会给进程内的线程分配一个线程的局部缓存，线程内的小对象就是在这个线程本地缓存（thread-local cache）中分配的。由于是线程本地的缓存，所以可以进行无锁操作，效率得到很大的提升。而TCmalloc管理的虚拟内存中的堆，称为中心堆（central page）。当线程本地缓存不足的时候也会从中心堆批量获取内存块来作为线程本地缓存，这个操作是需要加锁的。大对象也会从中心堆中分配。

内存对齐

C语言可以用offsetof函数来获取结构体成员的偏移量

基于Fedora32 64位系统

简介

现代计算机的内存空间是按照字节来划分的，但计算机系统对于基本类型的数据存放在内存的地址有限制。计算机系统会要求这些数据的存放起始地址是某个数的倍数（2、4或8等等），这个就是内存对齐。而CPU在访问数据时候，并不是按照一个字节一个字节来访问的，而是按照多个字节组合作为访问单位来周期性性访问。这个多个字节组合一般是：2个字节、4个字节、8个字节甚至是16，32个字节。而这些访问的单位被称为内存存取粒度。

CPU存取示意图

为什么要对齐

假设CPU存取粒度为4，要读取一个int32（4个字节）的变量，根据内存存取粒度为4的情况，每一个周期读取数据都要读取4个字节。假设现在没有内存对齐，并且这个int32类型的变量数据随意存放，假设它的存储起始地址是1，那么这个数据占据的内存地址就是1，2，3，4。现在要从0地址开始读取，一个周期读取4个字节，包括0，1，2，3。但是int32变量的数据还包括地址为4的数据，那么就需要进行迭第二个周期读取，读取包括4，5，6，7。然后将不需要的字节剔除，就是把0，5，6，7这个四个地址的字节剔除，最后剩下的1，2，3地址的数据块和地位是4的数据块组合得到int32变量的数据。

数组和链表

数组

数组是一种数据类型相同，长度固定，顺序存储的数据结构。也就说初始化的时候，会分配一块固定大小的内存，元素的存储时连续的。

优点

  1. 在下标范围内支持快速随机读取和更新数据，时间复杂度为O(1)

缺点

  1. 在插入和删除元素的时候需要移动元素位置。时间复杂度为O(n)
  2. 长度固定，想要存储更多元素的时候需要扩容

数组结构体声明
因为需要扩容，所以不直接使用数组，而是封装成一个结构体

Redis之持久化

简介

Redis在大多数情况下在内存中运行，数据是临时性的。但是Redis也提供将数据写入磁盘保存的方式，将数据写入磁盘保存的过程称为持久化。持久化的作用是在Redis宕机之后导致数据丢失的情况下，可以快速恢复之前存储的数据。在Redis中持久化有两种方式：一种是快照（RDB），另一种是只追加文件（AOF）。RDB持久化方式会定时将数据写入指定的文件中（覆盖），而AOF持久化方式会在执行命令的时候将写命令复制到磁盘里面（追加）。

RDB持久化

RDB持久化会在某个时间点（在redis.conf配置文件中指定）将内存中的数据的副本进行存储。存储的方式可以是写入本地磁盘文件中，也可以是将副本复制到其它Redis服务器中。不过这种方式有一种弊端：如果在持久化未完成的时候，由于某些原因而导致Redis进程停止，那么在上一次持久化完成之后的操作得到的数据就会丢失。。。比如：第一次RDB持久化完成之后接着执行写命令，过一段时间后进行第二次持久化。在第二次持久化的过程中出于某些原因Redis进程停止，那么第二次持久化操作就会失败。快照文件只会保存着第一次持久化的数据，在第一次持久化之后的数据都会丢失。

RDB持久化的方式

第一种是执行显示手动执行BGSAVE或者SAVE命令，这两个命令都会创建一个快照。不同的是，BGSAVE命令会使得Redis主进程派生出一个子进程进行快照的创建，主进程继续执行客户端的命令。而SAVE则是在Redis创建快照完成之前都不会处理客户端的命令。这么一看似乎BGSAVE命令比较好一些，但是这个命令有一个致命的问题。那就是在数据非常大的时候，执行BGSAVE命令会因为创建子进程而导致Redis停顿，数据量越大停顿越久。而在这种情况下，SAVE命令表现得要比BGSAVE命令好得多，因为不需要创建子进程，没有子进程来争抢资源，创建快照得时间要比BGSAVE命令快一些。

关于BGSAVE创建子进程导致停顿得问题，我的猜想是：当fork出一个子进程的时候，需要为这个子进程创建进程控制块，进程所持有的数据集等等数据，将父进程的进程映像复制一份。因为父进程所持有的内存块过大，导致复制过程缓慢。具体的理解还得详细阅读阅读操作系统原理和源码才知道。

第二种是通过配置文件预定义持久化行为，配置文件是redis.conf。持久化策略解释：save [length] [times]，这个意思是指，距离上一次持久化完成开始算起，当length秒内有times写入就会触发BGSAVE命令。比如：“save 900 1”是指距离上一次持久化完成到现在900秒内有1次写入就会触发BGSAVE命令。如果配置了多个持久化策略，那么只要满足其中一项就会触发BGSAVE命令。
