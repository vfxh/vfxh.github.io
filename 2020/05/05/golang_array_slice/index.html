数组

数组是一种具有相同数据类型的零个或多个，并且有固定长度的元素序列，Golang中数据的长度也是数组类型的一部分。不同长度的数组即使元素数据类型相同也是不同的数组类型。

数组的初始化
一维数组在初始化的时候可以省略长度，用…替代，数组的长度取决于初始化元素序列的长度。对于多维数组，只能第一维省略长度。如果使用索引指定初始化值，那么数组的长度由最大的索引加上1。但是，数组索引初始化不能有多个相同的索引初始化。

对于==，!=这些操作符，取决于数组元素的数据类型是否支持。如果数据类型支持这些操作符，那么数据也支持

map类型的对象只能和nil值相比较。原因是map类型的变量是一个散列表的引用，即使类型相同，数值也相同。但是它们引用的底层散列表并不是相同的，所以map变量之间不能比较。因此，map类型的数据之间不能比较。
在Golang中，数组函数传参是传值的方式。所以在另一个函数修改数组，影响的只是那个函数的局部变量数组，和传进去的实参无关。

slice

slice本质上不是一个动态数组，而是一个结构。这个结构有三个成员，分别是：len、cap、array。len是当前数组的元素数量，cap是当前数组的容量，array是一个指针，指向底层数组。

初始化

slice初始化可以引用现有的数组，也可以直接使用make函数。使用引用现有数组的方式初始化，slice对象的底层数组会和现有的数组是同一个数组。使用make函数初始化的方式，那么slice对象独占一个底层数组（不考虑赋值给其它slice的情况）。

字面量初始化

字面量初始化时直接指定slice类型和元素数量，初始化后的slice的长度和容量都是初始化时的元素数量，golang会自动创建一个底层数组。

另外需要注意一种情况，就是字面量初始化的时候元素列表为空，容量和长度都为0，但是不代表没有初始化，因为它已经引用底层数组了。这和slice声明类似，容量和长度都为0，但是没有引用任何底层数组。

打印结果

所以，判断一个slice对象是否是未初始化，那么就和nil比较，而不能判断长度是否为0。因为未初始化和已经初始化了的slice对象都有可能长度为0。另外，slice只能和nil直接比较，而不能进行slice对象直接相互比较，因为slice类型是间接引用的。

使用make函数初始化

在使用make函数初始化的时候，需要指定slice的类型，底层数组的初始化长度以及可选的初始化容量。当不显示指定初始化容量的时候，初始化容量就等于初始化长度。初始化容量和初始化长度不为0的情况下，slice对象的len个元素的值为对应类型的零值。而指定初始化容量不为0并且指定初始化长度为0时，slice对象的元素为空。

调用make函数创建slice的时候，会调用runtime.makeslice函数。runtime.makeslice函数接收三个参数：元素类型，slice的初始化len和cap。然后申请指定大小的内存块，计算方式是：

内存大小 = slice元素类型大小 * 初始化的slice容量cap

runtime.makeslice创建的过程中，如果遇到以下情况就会出错：

  1. 内存溢出
  2. 申请的内存超过最大可分配的内存大小
  3. len小于0或者len大于cap
使用append

append是golang的内置函数，其作用是对slice进行追加元素，然后返回slice。append函数可以追加一个元素，也可以追加多个元素。在调用append追加元素的时候，如果追加的元素数量超过了当前的slice容量，就会重新分配一块更大的内存。这也是为什么slice在未初始化的情况下还能使用slice追加元素的原因。

赋值和比较

slice引用类型，也就是一个slice对象间接引用了一个底层数组。所以，slice对象之间用‘=’的赋值只是类似于指针一样赋值。复制之后，两个slice对象都是引用同一个底层数组，也就是说：无论这两个slice对象之间哪一个修改了元素值，另外一个都能看得见。这种赋值被称为浅拷贝。

slice扩容

当append追加元素遇到底层数组空间不足的情况就会出发扩容机制，重新分配内存，拷贝数据，然后相关指针指向新的内存并返回slice对象。也就是说，append追加元素之后，可能返回的slice对象的底层数组和原来的slice对象的底层数组是同一个，也可能返回的slice对象的底层数组不是同一个。所以需要用相同的slice类型变量接收append返回的slice对象，保证在追加元素之后能够正确处理。就比如：*sli = append(sli, 1,2,3)向变量sli引用的slice对象的底层数组追加元素，然后sli变量引用append返回的slice对象，即使期间发生过扩容也可以使用sli来正确访问。如果是这样sli2 = append(sli, 1,2,3,4)*，那么发生扩容之后，sli变量引用的还是原来没有扩容的slice对象，而sli2变量才是引用扩容之后包含旧元素和新元素的slice对象。这也就是为什么append返回结果和传入的slice变量是同一个原因。

slice扩容原理

当使用内置函数append对slice对象的底层数组进行追加元素的时候，如果遇到底层数组容量不足的情况会调用runtime.growslice函数重新分配新的内存块，然后将数据拷贝到新的内存块当中。runtime.growslice参数接收三个形参，元素的类型（其中size成员是在计算机中占的字节数量），旧的slice对象以及一个期望的最小增长容量cap。这期望往的容量cap是go运行时估算的，由运行时决定。

在学习slice的扩容策略之前查资料的时候很多博客都只是讲了一部分，就是在空间不足，需要扩容的时候：对于元素个数小于1024的旧按原来容量的两倍扩容，如果大于或等于1024的情况就按原来容量的1.25倍扩容。如果runtime.growslice代码只有下面这几行，也没什么不对，恰巧也有那么一些情况在程序执行之后就是按照这个策略来扩容的。

这一部分代码主要做的事情就是根据运行时估算的新的slice容量和原来的旧slice容量比较，然后确定出初步的新的slice容量。

在初步确定新的slice容量之后，紧接着就是对确定的新容量根据go的内存分配规则进行调整。容量的调整会根据四种不同情况来进行不同的调整策略：

  1. 元素大小为1个字节
  2. 元素大小为系统指针占用的字节大小，不同平台有不同的大小，64位平台占8个字节。
  3. 元素大小位2的指数倍
  4. 除了上面三个情况之外的任意情况

runtime/msize.go

简单提一下，go内部内存管理的基本单元称为span，span可以分成67个规格。span可以分割为很多个object，object就是要分配给go的对象使用。其具体计算的算法很复杂，不过也有写死的表查询。

class：表示span规格的索引
bytes/obj：同一个span规格中object占用的字节数
bytes/span：span的大小，单位字节
objects：一个span可以分割成多少个object
tail waste：span尾部浪费的字节数
max waste：最大浪费的内存比例

runtime/sizeclasses.go

runtime/sizeclasses.go

runtime.roundupsize函数会根据传进去的参数通过计算之后来获取class_to_size数组的下标，返回返回对应的下标的值。比如传进runtime.roundupsize函数的参数size值是5，那么size+smallSizeDiv-1)/smallSizeDiv = 1.5。golang的无类型常量整数相除之后向下取整就是1，所以取数组size_to_class8的第二个元素就是1，再以1为索引取数组class_to_size的元素就是8。因此runtime.roundupsize函数最后会返回8。
所以，最后扩容之后的slice对象容量就是8。

倍数增长的例子

使用gdb调试，查看append触发增长的时候传入参数的详情。进入growslice函数的时候，传入估算的容量是3。从之前的例子来看，这里传入的容量是当前元素个数加上追加元素个数的总和，我先这么假设是这个情况。

后面就不一步步调试里，根据上面调试的套路。int类型占64位CPU架构上是8个字节，所以进入的分支应该是：et.size == sys.PtrSize。旧slice的cap是2，所以新的slice的初步容量是4（旧cap的两倍，比预估的cap->3要大，所以取4），因此newcap是4。所以这一行代码capmem = roundupsize(uintptr(newcap) * sys.PtrSize)中uintptr(newcap) * sys.PtrSize这里计算出来的数值是32。得到32之后传入roundupsize函数，经过转换之后最终得到的数值也是32（可以根据传进去的32计算出size_to_class8数组的索引，再根据这个索引找到相应的值作为下标，再去获取class_to_size数组中对应的值）。而在这个分支处理之后就可以最终确定最后的容量，就是根据分支里面的这一行代码*newcap = int(capmem / sys.PtrSize)*来确定的。带入数值之后得出，newcap等于4。这就是最后确定的容量，是扩容之前的slice容量的两倍。

runtime.growslice

但是，为什么这里会是两倍，而上文的某些例子不是呢？答案就是在slice调用growslice函数的时候传入的参数cap在作怪。这个cap就是现有元素加上要追加元素的个数，这个参数影响着slice的增长策略之一。比如上面的例子，假如要修改这个cap参数，只需要调整追加元素个数或者初始化个数即可。将第二次追加的元素扩充到三个，那么传入growslice函数的cap参数值就是5。

修改后非倍数增长

用gdb查看，很明显假设是对的。传入growslice的cap参数就是当前已有的元素个数加上追加的元素个数的总和。

修改追加的元素个数之后的容量。已经发生改变了，不再是按照旧容量的两倍来曾长的。

总结
slice的增长并非简单的在容量小于1024的时候按照两倍容量来曾长，大于1024的时候按照1.25倍来增长，虽然某些情况是这样的。准确来说，slice的增长很难预测得到。因为其中涉及到golang内存管理的部分内容，而且内存管理那部分也是写死的分配策略，slice中增长所需要的内存大小都是查表来决定。不同数据类型有不同的分配内存策略，现有的元素个数和追加的元素个数的总和，这些都是影响最终新slice容量的因素。所以单单用按照两倍或1.25倍的增长策略看待我觉得不大合适。

slice增长简易流程

  1. 调用growslice函数，参数有：表示slice元素数据类型的结构体et，旧slice对象，现有元素个数和追加元素个数的总和cap
  2. 比较cap和旧slice容量的两倍之间的大小，选择其中较大的来初步确定新slice的容量
  3. 根据元素数据类型选择不同的内存分配策略，调用roundupsize函数来调整申请内存的大小，使之符合内存管理的规格
  4. 申请内存，迁移数据，构建新的slice对象并返回

总结

区别：

  1. 数组是长度固定的，只能修改元素而不能增加或删除元素。默认数组传参是值传递，也就说修改形参并不影响实参。
  2. slice长度不固定的，可以增长，通过申请更大的内存块来保存增加的元素。slice传参默认是引用传参，修改形参的元素实参是可见的。但是追加元素有可能会引发扩容，元素迁移。

联系：
slice是引用底层数组的来实现的，如果当前底层数组容量不够，就会引用更大的底层数组。
