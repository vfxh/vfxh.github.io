Redis之持久化

请设置文章作者
Redis

发布于：May 23, 2020
次浏览

简介

Redis在大多数情况下在内存中运行，数据是临时性的。但是Redis也提供将数据写入磁盘保存的方式，将数据写入磁盘保存的过程称为持久化。持久化的作用是在Redis宕机之后导致数据丢失的情况下，可以快速恢复之前存储的数据。在Redis中持久化有两种方式：一种是快照（RDB），另一种是只追加文件（AOF）。RDB持久化方式会定时将数据写入指定的文件中（覆盖），而AOF持久化方式会在执行命令的时候将写命令复制到磁盘里面（追加）。

RDB持久化

RDB持久化会在某个时间点（在redis.conf配置文件中指定）将内存中的数据的副本进行存储。存储的方式可以是写入本地磁盘文件中，也可以是将副本复制到其它Redis服务器中。不过这种方式有一种弊端：如果在持久化未完成的时候，由于某些原因而导致Redis进程停止，那么在上一次持久化完成之后的操作得到的数据就会丢失。。。比如：第一次RDB持久化完成之后接着执行写命令，过一段时间后进行第二次持久化。在第二次持久化的过程中出于某些原因Redis进程停止，那么第二次持久化操作就会失败。快照文件只会保存着第一次持久化的数据，在第一次持久化之后的数据都会丢失。

RDB持久化的方式

第一种是执行显示手动执行BGSAVE或者SAVE命令，这两个命令都会创建一个快照。不同的是，BGSAVE命令会使得Redis主进程派生出一个子进程进行快照的创建，主进程继续执行客户端的命令。而SAVE则是在Redis创建快照完成之前都不会处理客户端的命令。这么一看似乎BGSAVE命令比较好一些，但是这个命令有一个致命的问题。那就是在数据非常大的时候，执行BGSAVE命令会因为创建子进程而导致Redis停顿，数据量越大停顿越久。而在这种情况下，SAVE命令表现得要比BGSAVE命令好得多，因为不需要创建子进程，没有子进程来争抢资源，创建快照得时间要比BGSAVE命令快一些。

关于BGSAVE创建子进程导致停顿得问题，我的猜想是：当fork出一个子进程的时候，需要为这个子进程创建进程控制块，进程所持有的数据集等等数据，将父进程的进程映像复制一份。因为父进程所持有的内存块过大，导致复制过程缓慢。具体的理解还得详细阅读阅读操作系统原理和源码才知道。

第二种是通过配置文件预定义持久化行为，配置文件是redis.conf。持久化策略解释：save [length] [times]，这个意思是指，距离上一次持久化完成开始算起，当length秒内有times写入就会触发BGSAVE命令。比如：“save 900 1”是指距离上一次持久化完成到现在900秒内有1次写入就会触发BGSAVE命令。如果配置了多个持久化策略，那么只要满足其中一项就会触发BGSAVE命令。

AOF持久化

AOF和RDB的不同之处是：RDB持久化方式是定时将写入的数据持久化到磁盘文件当中，而AOF则是把将被执行的命令追加到文件的末尾。RDB持久化的对象是数据，AOF持久化的对象是命令。所以，AOF持久化恢复数据的方式是将文件中的命令全部重新执行一次，就可以恢复之前的数据集。

同步策略

向磁盘写入文件的时候，首先会将数据写入到操作系统的缓冲区，然后由操作系统决定何时写入到磁盘。不过可以通过配置文件来控制这个同步的方式，有三个选项，不同的选项有不同的同步策略。

appendfsync选项

关键字 说明
everysec 每秒执行一次同步，显示将多个命令同步到磁盘
always 每个命令都同步到磁盘，这会降低Redis的速度
no 由操作系统决定什么时候同步

重写/压缩AOF文件

AOF方式有一个弊端，就是不断将客户端执行的命令追加到文件当中，会导致文件过大，甚至会耗尽磁盘空间。另外在Redis恢复数据的时候，由于文件中存储的命令过多，可能需要很长时间才能恢复过来。但是有些情况是不需要存储那么多次命令的。比如，对一个存储在string数据结构的整数型计数器进行INCR操作100次，备份的情况下只需要保存最后一次修改的值就可以。但是默认AOF情况下需要记录一百次，这个是完全没必要的。实际上可以使用SET key value这个命令就可以保存当前值了，其余99条命令都是多余的，可以移除。

对此，可以使用BGREWRITEAOF命令来生成一个新的AOF文件，删除那些冗余的命令，使得AOF文件尽可能小。BGREWRITEAOF命令和BGSAVE命令类似，也是通过派生出一个子进程来进行AOF文件重写。如果AOF文件过大，在重建完AOF文件之后删除旧文件的时候就会导致系统挂起，处于瘫痪状态。
