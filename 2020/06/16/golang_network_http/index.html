C/S和P2P的区别

在同一时间，P2P可以既是client也可以是server，而C/S只能是在不同的时间段内互换client和server这两个角色。

进程间通信

进程间通信（IPC）只是一组进程间进行数据交换的行为。进程的IPC是由操作系统控制的。操作系统中的IPC是运行在同一台主机上的不同进程。通信网络中的IPC可以是同一台主机上的一组不同进程，也可以是不同主机（可能运行的操作系统也不同）之间的一组进程。
网络IPC是通过一个叫做套接字（socket）的软件接口来作为报文交换的入口点和出口点，而这个套接字的就是IP和port的组合体。IP的作用是定位实体所在的位置，port的作用是定位报文交换的进程。
应用层的进程可以控制应用层的一切，但对于运输层的控制是有限的。比如可以选择用哪种运输层协议，TCP或者UDP。以及缓冲区的大小，报文段的长度大小。

HTTP相关

HTTP即超文本传输协议，目前主要基于运输层的TCP协议。一个HTTP的请求和响应过程的简单：先建立起TCP连接，然后进行数据交互，最后断开TCP连接。TCP在建立连接的时候会进行三次握手。在最后一次握手的时候，也就是建立起连接状态的第三次client向server发送的报文，会把HTTP请求报文作为第三次发送的报文发送到server中。

示意图

对于HTTP/1.0，每一个HTTP请求响应的结束都会断开TCP连接。如果想再次进行通信，就需要重新建立TCP连接。每次建立TCP连接都会带来额外的开销，就是分配TCP发送/接收缓冲区和相关变量。而且，每次重新建立起TCP连接到HTTP接收到server的响应，这都要花费两倍的RTT（Round-Trip Time：往返时间，指从C端发送数据到S端，然后S端返回响应，C端收到S端的响应的一个完整过程所花费的时间）。这就是HTTP/1.0的缺点。

HTTP/1.1解决了这个问题。就是在一条已经建立起的TCP连接上完成了HTTP请求和响应的过程后不断开这条TCP连接，而是保持一段时间的连接状态。这种连接方式叫做持续连接。后面相同的客户端再次进行HTTP请求的时候就不用再次建立TCP连接。这个时候，一次HTTP请求和响应的过程所花费的时间就下降到近似一个RTT，还减少了由于TCP连接建立的时候分配缓冲区和相关变量的开销。

建立连接通信结束后就断开连接叫短连接，而通信结束后还继续保留一段时间的叫做长连接或者持续连接

（！！！！这里要使用Linux的TCP连接工具实际操作下！！！！）

HTTP/1.1中的持续连接有两种工作方式，一个是非流水式，一个式流水式。

非流水式

非流水式的工作方式下，client发送一个HTTP请求之后，只有等到这个请求的响应返回之后才能再次发送HTTP请求。也就是说，在发送请求到收到响应这一段时间TCP连接是处于空闲状态的。虽然每请求一个资源花费的时间是一个RTT，比HTTP/1.0的短链接的2xRTT要高效率，但是还是会浪费点请求到响应的那一段中间的时间。

流水式

相对于非流水式的工作方式，流水式允许client在发送一个HTTP请求之后，不需要等到响应就可以再次发送HTTP请求。这种工作方式将非流水式的那端请求响应的中间空闲时间充分得利用起来，提高了效率。在多个连续的HTTP请求到server之后，server会根据请求返回相应的HTTP响应。不过需要注意的是：返回的HTTP响应可能不会按照请求的顺序到达，这其中涉及到server端处理的时间问题以及网络的时延问题。总的来说就是，HTTP/1.0流水式的工作方式要比非流水式的工作方式要高效。

特性

  1. 面向事务
    HTTP事务是指一系列信息的交换，这个交换的过程是不可分割的整体。要么全部交换完成，要么一次都不交换。

HTTP一次完整的请求响应过程

  1. 无连接
    HTTP协议本身是无连接的，它依赖于运输层的TCP协议提供可靠的，按序到达的，无差错的服务。

  2. 无状态
    所谓无状态是指，HTTP协议本身并不要求server保存有关于client的任何信息。每一次client发来的请求server都不知道client的身份。比如，client向server第一次请求server，然后server做出响应。相同client再次发送相同请求server，server都会向第一次一样对这个请求做出相同的响应。也就是说，HTTP server并不关心谁是谁，只要请求符合要求就会返回响应。

URI和URL

URI： (uniform resource identifier) 统一资源标识符，用于唯一标识互联网中的某个资源。这个资源可以是单个，或者多个的集合。

URL： (uniform resource location) 统一资源定位符，用于表示资源的地点。URL是URI的子集。

怎么理解URI呢？关键点就在于”唯一“这两个字，无论什么东西，只要有能够把这个东西唯一标识出来，那么就可以说这个标识是URI。比如一台主机上的某个资源（文档，或者某张图片）。这个资源有唯一的标识，那么能够找到这个资源的一个字符串就是URI。比如：ftp://ftp.is.co.za/rfc/rfc1808.txt、http://www.ietf.org/rfc/rfc2396.txt 这些都可以叫做URI。那么URL又是什么东西呢？所谓的URL就是一个可以获取到这个资源的路径，也就是位置或地点。比如上述URI中的 ftp://ftp.is.co.za/rfc 或者 ftp://ftp.is.co.za 。所以说，URL是URI的子集。相对URL是从URI去掉主机信息部分之后的字符串，比如/rfc/rfc1808.txt。如果一个URI不包含单一的资源，比如 http://www.baidu.com 那么相对URL就是 / 。简而言之URL是URI的一种表示。

通用URI格式：

方法 说明
协议名 诸如：http、ftp、https等等表示协议的字符串
用户名和密码 认证信息，可以省略
主机名 可以是域名，也可以是IP
端口号 服务端监听的端口号，如果是http协议，那么默认80端口，可以省略
路径 类似于Linux系统的路径，/ 分隔
查询参数 可选，用于请求服务端返回的特定数据参数
片段ID 标记获取到资源中的子资源，比如文档内的某个位置

HTTP报文格式

HTTP协议式面向文本的协议，所以报文的字段是一些ASCII码串，码串的长度式不固定的。

请求报文通用格式

HTTP请求报文由三个部分组成，分别是：请求行，首部行和实体主体。在一些请求报文中，实体主体这部分是省略的，如GET方法等。首部行和实体主体之间用一个空行分隔开。

请求行
这一部分可以用来区分请求报文和响应报文。请求行由三个字段组成，各个字段之间用空格隔开，最后是回车和换行结尾。第一个字段是方法，表示该报文使用的是什么方法（GET或者POST等等）。第二个字段是URL，一般使用相对URL，然后在首部行给出目的主机的域名或IP部分。第三个是HTTP协议版本，如：HTTP/1.0，HTTP/1.1等等。

首部行
首部行可以有多个，通常是描述报文或者客户端的一些信息，还有控制信息等等。也可以自定义一些需要的字段和值

实体主体
就是需要传到server的具体数据，可以省略。

HTTP方法
方法，实际上就是要求server做出相应动作的命令。比如想要获取一个资源，那么可以用使用GET方法，然后server端返回相应的资源。

当然这只是一个规则，遵不遵守就不一定了。比如你用GET方法请求资源，结果server删除了这个资源，干这种事的人又不是没有…

方法 说明
GET 请求server返回指定的资源
PUT 请求在server创建一个资源或者更新一个资源
POST 请求在server创建一个资源
DELETE 请求删除在server的一个指定的资源

HTTP请求实例

分析
请求行中的第二个字段是一个相对URL，只有一个“/”，因为请求的URL只有一个域名：www.baidu.com。假设请求的URL是：https://www.baidu.com/paths/source.md。那么在请求行中的URL字段就是：/paths/source.md。在首部行中的Host字段是域名或者IP:port（如果使用的ip和port访问的话），不包括完整URI的前缀“https://”。在这里，Host就是首部字段，www.baidu.com 就是这个字段的值。其它首部字段和值也是类似的。而因为GET方法不携带实体主体，所以在报文的最后只有一个空行。实体主体字段为空。

响应报文通用格式

相应报文和请求报文的格式类似，不同的是报文的第一部分。响应报文的第一部分叫做状态行，同样有三个字段组成，分别是HTTP版本，状态码，端语。状态码三个数字组成，有很多个。有些是熟知的，如200，201，401等等，它们都有固定的含义。当然后也可以自己定义，不过要遵循HTTP的规则。第三个字段是短语，其作用是对返回的状态进行简短的说明。比如：Accepted，Not Found等等。后面的和请求报文差不多，实体主体也是可选的。

HTTP状态码大致可以分为五类，下面是大概的说明。

状态码 说明
1XX 通知信息，可能server正在处理
2XX 表示成功
3XX 表示重定向
4XX 表示客户端请求错误，参数或格式不正确
5XX 表示服务端错误
