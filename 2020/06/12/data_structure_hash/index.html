散列表

Posted on 2020-06-12 Edited on 2021-05-23 In 数据结构

基本概念

散列表

散列表也叫哈希表，是一种能够实现将多个键映射到多个值的数据结构，其添加，删除，查找等操作都是常数时间完成。其中，键和值的映射关系是通过散列函数来完成的。理想的散列表是一个固定大小的数组，不同的键经过散列函数的计算之后能够均匀分布到不同单元的散列表项当中，但这只是在理想的状态下。

散列函数

散列函数就是将任意长度的数据映射到固定长度数据的函数。散列表使用散列函数来计算键在散列表的索引，来找到键在散列表中对应的值。一个好的散列函数能够将不同的键均匀的映射到散列表的不同单元中，提高散列表利用率。所以，散列函数的好坏取决于其解决冲突的能力。但凡事都有利弊。一个好的散列函数通常会增加计算的开销，效率有所降低。所以，适当选择不是很完善的散列函数有助于提高效率。

取余法

最简单的散列函数是取余法，以键的长度对散列表的大小取余。对于键是字符串的形式，通常会先计算字符串的长度，然后根据字符串的长度对散列表的大小取余。一般来说，散列表的大小取素数会更好。取素数的散列计算方式：hash(key) = len(key) mod hashTableSize。如果散列表的长度不是素数，那么计算公式就是：hash(key) = (len(key) mod p) mod hashTableSize，p是一个大于hashTableSize的素数。

为什么取余法的散列表长度要取素数

简而言之就是为了减少冲突，使得键在散列之后能够均匀分布在散列表中

负载因子

负载因子也叫装填因子，或者其它什么因子。是反映散列表的装填程度的一个参数，负载因子越大，表示装填程度越高，发生冲突的概率也更大。其计算公式是：a = 散列表元素数量 / 散列表长度

再散列

当散列表到达一定程度的时候，重新申请一个比当前更大的新表（比如旧表容量地两倍）。然后将原来的表元素重新散列到新表上，这个过程就叫做再散列。再散列可以使得原来紧凑的表元素稀疏地分布在新表上，减少冲突的概率，提高效率。不过这个过程时很耗费时间，因为需要重新构建一个新的散列表，然后遍历旧表，将元素重新散列到新表上。再散列触发的时机有几个：一是当表元素的个数达到表容量的一般时再散列，二是当插入元素失败时再散列，三是当负载因子达到一定的值时再散列。相比较之下，时机三是这几个当中最好的。

散列冲突

理想中的散列表是不存在两个不相同的键散列在同一个散列表项上的，但现实就是现实。实际情况中散列表是有限长度的数组，而键是没有限制的。所以，不同的键有可能映射到相同的散列表单元中，这种情况叫散列冲突。

冲突示意图

解决冲突的方法有开放定址法，分离链接法等。

开放定址法

在开放定址法中，当一个键和另一个键发生冲突的时候，就需要查找下一个可用的位置，而不是覆盖住当前的位置。所以，使用开放定址法来解决散列冲突，就必须存在可用的位置。反之，插入元素失败。也就是说，只要表足够大，就会有存储元素的地方。不过缺点是效率不怎么理想。特别需要注意的是，使用开放定址法删除表元素时不能真的删除。而是标记这个元素已经删除了，但是数据还在，只不过在某些操作的时候忽略其存在（比如查找）。这种标记删除的方式叫做懒惰删除。这么做的原因是，一个键在插入表中的时候可能存在冲突。冲突之后需要查找后面的可用位置存储。假如一个冲突的元素存储的位置不在散列得到的那个位置上，而是经过查找后存储在偏移原来若干个位置的地方。那么在这种情况下删除了中间的某个元素，这个时候再查找之前存储的元素就不能正确查找了。有点绕，画个图帮助理解。

对于开放定址法，负载因子应该小于0.5。其原因让元素尽量稀疏地分布在表上，减少冲突地概率。避免发生严重地聚集现象，降低性能。

懒惰删除
指的是某个位置元素并不是真正地被删除，而是呈已经删除状态，实际数据还存在，就是”假装“被删除了。在开放定址法中，表项地数据不宜被删除，因为可能存在散列地位置和真正存储位置并不是同一个位置。如果有一个表项被删除了，那么后续的查找操作可能会失效。

就像下面的示意图，假设元素D散列到索引1这个位置，但是索引1的位置已经被A占用了。那么就需要往下一个位置找，一直找到索引为4的位置，这个位置可用，然后将D存储到这个位置上。后面删除了B，然后索引2的这个位置的数据被清空。而查找D的时候，首先散列到索引1的位置。发现不符合，然后往后找。但是B已经被删除了，这个位置为空。根据开放定址法的特性，查找到位置为空的情况就表明元素不存在。所以，D元素就没办法正确查找到了。这就是为什么开放定址法不能够真正删除元素，而要使用懒惰删除的原因。

当然除了懒惰删除还有别的方式，其中一种就是移动元素，就像数组中删除了某个元素之后需要将后面的元素向前移动一个位置。不过这种方式效率并不理想，每次删除元素都需要移动元素。如果表的长度很大，那么这种方法会导致效率急剧降低。相对来说，还是懒惰删除比较理想。

示意图

在开放定址法中，元素在表中的索引可以表示为：h(key) = (hash(key) + f(i)) mod Tsize。查找下一个可用位置的方法很多，最简单的就是线性探测法。

线性探测

在线性探测法中，函数f取f(i) = i。也就是在第i次探测的位置等于(hash(key) + i) mod Tsize。这个公式的意思是：一个键被散列到表的某一个位置，假如这个位置已经被占用。那么就递增i，从散列的位置开始依次探测下去，直到找到一个可用的位置。假如最开始散列得到的位置不是表开始的位置，那么在探测到表的尾部的时候还没找到可用位置，就从表头开始继续探测。一直探测到最开始散列到的那个位置。这个类似于循环队列一样的思想。

理论分析

假设有一长度是11地数组，使用散列函数是简单的取余法。有元素{47, 7, 29, 11, 9, 87, 54, 20, 30}，经过散列函数计算后得出各个元素在数组中最初的索引位置{3, 7, 7, 0, 9, 10, 10, 9, 8}。使用线性探测法后最终在数组中存放的位置就像下面示意图一样。只要出现冲突，就会从散列到的位置依次探测下去。探测的次数越多，元素距离原来散列到的位置偏移量就越大，查找的次数就越多。

示意图

最后元素在数组中的分布图

根据示意图中的分析，可以简单估算出元素的成功查找的平均长度和不成功查找平均长度。所谓的成功查找平均长度是指：一个存在于散列表元素从散列到的地方开始，它需要比较多少次才能找到这个元素的所在位置。而不成功查找平均长度是指：一个不存在于表中的元素，从散列到的位置开始，需要比较多少次才能判断出它不存在散列表中。一个元素不存在表中的判断方法就是，当从散列到的位置开始比较，一直到碰到一个空的位置时，就可以判断这个元素不存在了。这也可以用判断一个新的元素插入表中需要比较的次数。

成功查找的平均次数

ASL：result = sum(每个存在表中的元素的查找次数) / 散列表种元素个数
如上图的成功平均查找次数 = (1+1+2+1+1+1+3+5+8) / 9 = 2.44（近似）

不成功查找平均长度

针对于不存在散列表的元素，需要找多少次才能证明该元素不存在散列表里
比如22，散列函数计算出来的位置是0。那么索引0开始查找，需要找6次才能证明22不在散列表里。因为遇到空了，那么就可以证明22不存在散列表里。
所以，不成功平均查找次数就是散列表的每一个表项开始，需要查找多少次才能遇到空项的次数的和。
ASL：result= sum(从每一个表项开始比较，直到查找到空项的比较次数) / 散列表长度
如上图的不成功平均查找次数 = (6 + 5 + 4 + 3 + 2 + 1 + 1 + 10 + 9 + 8 +7) / 11 = 5.09（近似）

总结分析

根据上图的最终结果来看，会形成从索引0到索引4的连续区域以及从索引7到索引10的连续区域都被占用，即使表的大小比示意图中的大小还要大一些，有很多空位。但是还是有可能形成这样一段连续的区域被占用的情形，这样子的情况叫做一次聚集。即使散列函数被设计得很好，可以均匀地将元素分布在散列表上。但是无论如何都会有冲突的现象发生，这是不可能避免的。由于线性探测地特性（出现冲突之后依次探测，也就是一个单元一个单元地探测），一次聚集的现象也还是会发生的。

平方探测

使用平方探测法需要保证表的大小是素数并且至少有一半的空位（表的大小是元素个数的两倍，也就是负载因子最大是0.5，超过了就得扩容），这样才能保证新的元素插入。反之，即使只超过表大小的二分之一一个元素，也有可能插入失败（找不到可用的空位）。

平方探测法的实现只需要将原来的线性探测法的函数改改就可以了，根据探测通式：h(key) = (hash(key) + f(i)) mod Tsize。在平凡探测法中，一般取f(i)=i^2，但实际这个函数取的是二次函数，二次函数并非只是f(i) = i^2，这样子的函数所探测到的位置仅仅只是表的一半位置，一个解决的方法是：f(i) = i^2 - 1。i的取值范围是：0 < i ≤ Tsize-1。

平方探测法

平方探测法不扩容的元素散列分布。相对于线性探测法来说，平均冲突的次数少了很多。

扩容之后的平方探测散列分布，感觉没多大区别。可能用的例子不够恰当，不过还是可以看得出来，平方探测的聚集程度要比线性探测要小。

理想的状态下，表的大小应该是素数4j+3，其中j为整数，1 <= j。这种情况下，平方解决函数就是：f(i) = ±i^2。在这种条件下，就有可能探测到表的所有单元。不过这种解决方法会涉及到负数，负数取余得到的结果应该做特别的处理。比如：-6 % 23 = -6，这并不是想要的结果，并且数组的下标也不可能为0。所以，解决的办法就是：求模之后的结果再加上模，这就得到一个正数。比如：(-6 % 23) + 23 = 17。

总结，平方探测法相对于线性探测主要的区别在于：出现冲突之后，线性探测法是从冲突的位置依次向后探测，容易产生一次聚集现象。平方探测法解决了这个问题，在出现冲突的时候按照一个二次函数计算出来的位置继续探测，这样避免了线性探测的一次聚集现象。但也会产生另一种聚集现象，叫二次聚集。总的来说，平方探测要比线性探测表现要好。

分离链接法

分离链接法对于散列冲突的时候，其做法是保留散列到同一个值得所有元素到一个表中，通常是一个散列表。当然也可以是其它数据结构，比如B+树，红黑树等等。这里使用得单链表方式。所以，整个散列表就是数组加单链表的方式。每个数组的元素都包含一个单链表的头指针，指向一条链表。一个键所对应的值就保存在这个单链表中，如果出现散列冲突，那么就插入一个结点，形成一条单链表。
单链表的长度不宜过长，否则在执行查找操作的时候扫描一整条长长的链表，耗费的时候很多。

数组+单链表的分离连接法示意图（省略散列函数）

分离链接法的一些操作

  1. 插入键值对

    在向散列表中插入键值对的时候，首先通过散列函数对键进行计算来得出键在散列表中对应的位置。然后将和键对应的值插入该位置关联的链表中，即使不同的键计算出的位置相同（也就是冲突）也无关紧要，只需要将对应的值插入到链表当中就可以了。

  2. 删除键值对

    同样需要通过散列函数来计算键在散列表中的位置，然后遍历这个位置关联的链表。找到该键值所在的那个结点，就将其删除。如果不存在，要么什么都不做，要么抛出错误，结束程序。

  3. 根据键查找值

    通过散列函数来计算出键在散列表中的位置，然后遍历链表，找到和找不到的处理依赖于具体实现。

  4. 根据键修改值

    根据键找到对应的链表节点，然后更新这个键对应的值。如果找不到，那么就插入这个键值对。

为什么要扩容

散列表始终是有大小的，当散列表中的元素到达一定程度的时候需要扩容，来满足更多的键值对存储而不降低效率。对于分离链接法来说，散列表扩容的时候需要申请一块更大的内存块，然后更新其相关数据。更重要的是，扩容之后需要重新散列，将原来的键值对重新散列存储到新的结构当中，使得原来分布非常紧密的数据单元变得稀疏，降低冲突的几率。同时也会使得原来的链表长度减短，提高效率。这时候，原来的结构就会出现改变。有可能原来存储的位置在重新散列之后就不同了，这个依赖于具体的散列函数实现。扩容的依据是根据很多因素来决定的，其中负载因子就是一个很重要的因素，当负载因子大于或等于某个数的时候就会触发扩容机制，进行扩容操作。对于分离链接法来说，如果不扩容出现的情况就是每一个散列表单元所关联的链表就会很长很长，那么在这种情况下进行查找或更新操作所耗费的时间会非常多。最极端的情况下会和普通单链表一样，遍历整条链表来达到操作的目的。这种情况下，散列表的优势就不复存在了。

散列函数

使用分离链接法实现的散列表，散列函数使用的算法至关重要。一个通常的办法是根据散列表的长度对key（这里假设key是字符串）的长度取余，也就是：length(key) / tableSize。但是这种方法也有一定的缺陷，假设有一种情况，就是所有的key的长度都相同，但是内容不同。那么使用取余的方法就会造成所有的键都会散列到同一个位置上的情况，只会使用到一条链表。最后的情况就是这条链表会变得很长，而其它散列表项没有被使用，最终散列表退化为单向链表，效率急剧降低。

总结

散列函数的处理算法有很多，以后有时间再慢慢研究。从上面的分析来看，分离链接法主要消耗的时间在散列函数计算以及链表的遍历上，这两个部分设计得好会使得效率得到很大的提升。一个好的散列函数使得键计算出来的位置能够均匀地分布在散列数组上，链表地长度也相对较短。不过，提高效率的同时，也会增加碰撞的概率，使得链表长度增加。所以，负载因子这个参数很重要。它表示散列表的装填程度，当负载因子变得很大得时候就表明冲突概率增加，这就需要扩容，重新散列分布使得散列表变得稀疏，提高效率。简而言之，在散列函数设计得相对较好得情况下，表的大小不重要，重要的是负载因子，这也就是以空间换时间的一种做法。
在分离链接法中，其实也可以使用其它数据结构体来代替单向链表，以增加查找的效率，比如红黑树什么的。当然，我没研究到。
