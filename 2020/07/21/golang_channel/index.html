channel

和slice，map等类型一样，channel也是一种引用类型，其主要用于goroutine之间的通信。一个channel类型变量引用一个runtime.hchan底层结构，这个结构是channel运行时的表示。

数据结构

在创建一个channel的时候，如果是无缓冲channel，比如：ch := make(chan int)，那么runtime.hchan.dataqsiz为0，runtime.hchan.buf为空指针。如果是有缓冲channel，比如：ch := make(chan int, 1)。那么runtime.hchan.dataqsiz则是1，runtime.hchan.buf则是指向一块能够容纳1个int型变量的内存。这块内存实际用于数组，为了能够复用空闲的位置，这个数组被设计成循环队列。不过创建有缓冲channel却不分配缓冲区内存的情况也有，比如元素类型是struct{}，这个就不会创建缓冲区。 如果忽略元素类型为struct{}这种不占用内存的元素类型，可以总结为：有缓冲和无缓冲的channel实际区别就是，在创建一个channel的时候是否有缓冲区，也就是用于缓存待发送或接收元素的循环队列。当然这个总结有瑕疵，但总体大致是这样。

创建

创建channel是通过调用runtime.makechan函数来创建的，这个函数接收两个参数。第一个是元素类型，第二个是缓冲区大小。runtime.makechan会根据元素的类型和大小来确定是否分配缓冲区。比如元素类型为struct{}的情况，即使创建的channel是有缓冲的channel，runtime.makechan也不会分配缓冲区内存。对于不包含指针的元素类型，创建有缓冲channel的缓冲区是紧跟在runtime.hchan结构的内存后面。而对于包含指针的元素类型，则是单独分配内存空间。

收发

上文有提到过，channel分成有缓冲和无缓冲两种形式。无缓冲channel用于goroutine之间的同步，有缓冲则是用于异步。无论是有缓冲还是无缓冲的channel，在未初始化状态地channel进行发送或者接收操作都会阻塞。并且，关闭一个未初始化的channel会引起panic。

runtime.sudog中的runtime.sudog.elem成员是指向在channel传输的数据项的指针。runtime.g的runtime.go.param成员，用于唤醒当前阻塞的goroutine参数。在发送操作的情况下，发送者会获取正在阻塞的接收者，此时接收者是runtime.sudog的打包状态。然后发送者会将待发送的数据项复制到待接收者runtime.sudog.elem指针指向的内存块。接着发送者会从待接收者的runtime.sudog中获取真正阻塞的goroutine，并将goroutine的runtime.g指针指向这个runtime.sudog结构，最后会唤醒这个阻塞的goroutine。最后，唤醒之后的goroutine获取channel中的数据就是从runtime.g.param指向的runtime.sudog的elem指针指向的内存获取的。

写入（发送）

channel的发送操作 c <- x 编译期间会转换成调用runtime.chansend1函数，这个函数内部会调用runtime.chansend函数。runtime.chansend函数的形参中，c表示一个用于通信的channel，ep是指向在channel传输的数据项的指针，block表示是否阻塞表示，值为true时表示需要阻塞，callerpc是调用者的程序计数器，包含下一条要执行指令的地址。另外，这个函数有一个返回值。在用户程序层面似乎用不到这个返回值，返回true表示数据项正常写入channel。

总体概括

runtime.chansend函数大致可分成几个部分。首先是判断channel是否已经初始化，如果是未初始化状态执行的发送操作，当前goroutine就会进入阻塞状态，并将一直会阻塞下去。runtime.gopark会将一个goroutine从运行态切换到阻塞态。另外，如果向一个已经关闭的channel写入数据，那将会引发panic。下面是runtime.chansend函数的大体流程：

  1. 无论是有缓冲channel还是无缓冲channel，只要存在因为等待发送goroutine将数据项写入channel而阻塞时的接收goroutine，runtime.chansend就会调用runtime.recvq.dequeue方法获取等待的接收goroutine，然后调用runtime.send函数将数据项写入channel，并去唤醒因为等待发送者写入而阻塞的goroutine。被唤醒的goroutine会转为就绪态等待被调度，最终会执行这个等待的goroutine并接收数据。
  2. 如果当前channel是有缓冲的，并且没有接收者在等待链表里，而且缓冲区还没有。那么当前goroutine就把数据项写入到channel的缓冲区里面，并返回。注意，是直接返回，并不会阻塞。这也就是为什么有缓冲的channel在缓冲区还没有满的时候写入不阻塞的原因。
  3. 如果goroutine将数据项写入的是无缓冲channel，并且没有相应接收channel的goroutine在阻塞时，自身就会打包成runtime.sudog结构并发到channel中的发送阻塞链表里。同样，如果goroutine将数据项写入的是有缓冲channel，并且channel的缓冲区已经满了，那么自身也会打包放到channel的发送阻塞链表里面。

这部分是runtime.chansend函数最后的部分，也就是当前发送者阻塞的部分。如果channel是无缓冲的，并且还没有接收者在阻塞链表里面，那么当前goroutine就会执行到这里。如果是有缓冲的，但是缓冲区已经满了，那么同样会执行到这里。当前goroutine阻塞之后，唤醒操作是由接收者发起的。
最后一部分做的事情：

  1. 获取当前goroutine已经从当前P的缓存中获取一个runtime.sudog，然后将goroutine打包进runtime.sudog。runtime.sudog和goroutine相关联，并且完成打包之后放到channel的双向链表里面。
  2. 调用runtime.gopark阻塞当前goroutine，并调用runtime.KeepAlive保活用于传数据的数据所在的内存，避免被释放掉
  3. 阻塞的goroutine被唤醒，之后需要判断goroutine和runtime.sudog是否相对应。最后会将runtime.sudog归还给P的本地缓存中，方便复用。 将打包后的runtime.sudog放到阻塞链表尾部。（没错！当goroutine在channel上阻塞的时候是按顺序放到阻塞链表的尾部的，而获取的时候是从头部获取。也就是说，唤醒的goroutine就是当前阻塞链表中等事件最长的那个）

读取（接收）

channel的接收操作会根据不同的语法调用不同的函数。<- ch操作会调用runtime.chanrecv1函数，x, ok := <-ch和for x := range ch会调用runtime.chanrecv2函数。不过这两个函数都会调用runtime.chanrecv函数。

返回参数第一个应该是用于select选择，第二个应该是用于表示channel是否关闭

总体概括
runtime.chanrecv函数的第二个参数ep是一个指针，指向接收变量所在的内存块。比如：x := <-ch，那么ep就是变量x的地址。ep也可能为nil，就是不指向任何内存块，为nil时表示接收到的数据被忽略。比如：<-ch，在这种情况下，ep就为nil。另外，ep必须指向堆或者调用者的栈。如果从一个未初始化的channel接收数据的话，和发送操作一样都会阻塞。需要注意的一点是，当channel关闭之后，并且缓冲区已经全部被读取，那么就会清空ep指向的内存块数据，第二个返回参数返回false，表示channel已经关闭并且数据全部被读取。这也就是为什么从一个关闭的channel读取数据之后会返回零值的原因。比如：x, ok := <-ch，当channel关闭并且数据已经被全部读取之后，x就是对应类型的零值，ok的值为false。

和发送操作类似，接收操作大致可分成三个部分：

存在正在阻塞的发送者，直接接收部分
获取正在阻塞的发送者和发送操作中获取正在阻塞的接收者类似，都是从阻塞链表中的头部获取。不同的是，发送操作是从接收阻塞链表获取阻塞接收者。而接收操作是从发送阻塞链表中获取阻塞发送者。然后，调用runtime.recv函数将channel中的数据拷贝到ep指向的内存中，并去唤醒正在阻塞的发送者。对于runtime.recv执行的接收操作：

  1. 如果是无缓冲的channel，那么直接从阻塞的goroutine的打包状态runtime.sudog结构的elem成员指向的内存块读取数据（ep不为nil，则写入ep指向的内存块）。
  2. 反之，就是有缓冲的channel。那么就从缓冲区（循环数组）的第一个元素读取数据，也就是runtime.hchan.recvx的位置（ep不为nil，则写入ep指向的内存块）。

这里需要注意一点，当调用runtime.waitq.dequeue能够获取到一个不空nil的runtime.sudog的时候。说明两个问题：

  1. 当前channel是无缓冲的，所以会有一个正在阻塞的发送者。
  2. 当前channel是有缓冲的，并且有缓冲的channel的缓冲区已经满。如果缓冲区不满，通过有缓冲的channel通信的goroutine是不会阻塞的。所以，当channel是有缓冲的，那么必定缓冲区已经满了，才会有阻塞的goroutine。

因此，runtime.recv函数处理的就是无缓冲channel的接收操作，或者有缓冲channel并且缓冲区已经满的接收操作。

runtime.recv函数对于无缓冲的channel接收操作是直接将数据拷贝到接受变量的内存区（如果ep不为nil），然后唤醒发送者。而对于有缓冲的channel，注意，这里的channel缓冲区已经满了，满了之后自然就会有阻塞的发送者。对于缓冲区已经满了的channel接收操作：

  1. 从缓冲区的队头位置获取数据，将数据拷贝到接收变量的内存区域（如果有的话）。然后，将阻塞的发送者的数据拷贝到缓冲区已经被读取数据的位置。
  2. 调整recvx和sendx，这两个值相等，如果超过了循环数组的限制，那就需要调转回到数组的起始位置。然后去唤醒已经被拷贝发送数据到缓冲区的阻塞发送者，最后就是唤醒这个阻塞发送者。

runtime.recv函数对于缓冲区已经满了的有缓冲channel有些绕，下面继续借助具体例程和示意图来说明。例程中有五个发送goroutine，一个接收goroutine。为了方便理解有缓冲channel缓冲区满了的之后的接收操作，接收goroutine先是休眠两秒钟才开始执行接收操作。

实验例程

例程中首先启动五个发送goroutine，执行顺序未知。除此之外启动一个接收goroutine，并且休眠两秒钟，目的是让缓冲区写满再执行接收操作。在接收goroutine启动之前，已经有三个goroutine已经执行发送操作完毕并成功返回了，不会阻塞，它们的数据项被拷贝到了channel的缓冲区当中。但是之后的goroutine想要将数据写入channel缓冲区的时候就会阻塞，因为缓冲区已经满了。三个发送者完成写入操作之后缓冲区满了的channel结构示意图如下。

大约两秒钟过后，接收goroutine启动，从发送阻塞链表中摘取首节点也，就是最先阻塞的发送者。并将首节点（runtime.sudog）中保存的数据项拷贝到接收变量当中（如果接受变量存在的话），然后把即将被唤醒的接收者所持有的数据项写入到之前已经被读取的数据项在channel中的位置。更新sendx和recvx的值，并且唤醒一个正在阻塞的发送者。被唤醒的发送者从阻塞态转为就绪态，被调度运行之后它要么继续正常，要么成功退出。这里需要注意一点，唤醒一个发送者并将它的数据项写入channel的缓冲区之后，缓冲区仍然是满的。也就是说，在这个情况下如果有其它发送者尝试执行发送操作都会阻塞。然后打包成runtime.sudog放到发送阻塞链表中。

被唤醒的发送者数据写入队尾，因为它相对于之前已经成功写入的发送者来说是后来者，写入队尾才能保证先进先出的原则（FIFO）

第一次一次接收操作示意图：

接收goroutine接着执行第二次接收操作。发送阻塞链表还存在阻塞的发送者，于是摘取位于链表头部的发送者。和上一次接收操作一样，把即将要被唤醒的接收者的数据项写入队尾。然后更新recvx和sendx的数值，然后再去唤醒发送者。这里的channel缓冲区依然是满了的，所以如果有发送者执行发送操作仍然会阻塞。不过，假设没有发送者，那么发送阻塞链表就会是空的。因为到目前为止，所有的阻塞发送者已经被唤醒了。所以，如果再有接收操作，就不在调用runtime.recv函数，而是直接从缓冲区中的队头将数据读取出去。因为到目前为止，缓冲区中data存在的时间比da_1和da_2的时间还长。等到data被读取出去之后，recvx就回调转回到缓冲区的起始位置0。那么再次读取就是da_1了。这就是上面分析的，为什么要把阻塞发送者的数据项写入队尾的原因。

在分析其原理之后我们来看看例程的运行结果。这是其中一次打印的结果，由于goroutine的执行顺序不可知，所以每次运行例程得到的结果都不一样。但是可以确定的是，在接收goroutine开始执行接受操作之前并且缓冲区已经满了之后，其它执行发送的goroutine都会阻塞。在此次运行过程中，E、A、B三个发送者完成了发送，成功退出不阻塞。但是C和D这两个发送者执行发送操作就会阻塞。从打印结果中可以看到，在接收goroutine接收之前，channel中的缓冲区的数据项是3个，sendx和recvx都是循环数组的起始位置。然后执行第一个接收操作，阻塞的发送者的数据项被拷贝到缓冲区中，sendx和recvx往后移动，缓冲区还是满的。这里需要注意一点，接受操作和唤醒操作时分开的，所以可能在拷贝数据到缓冲区之后被唤醒的发送者还没开始执行，具体取决于goroutine的调度。然后进行第二次接收操作，和第一次接收操作情况一样。即将被唤醒的发送者的数据项拷贝到缓冲区当中，然后去唤醒发送者。最后，所以的发送者都被唤醒。程序不会一直阻塞，goroutine全部成功退出。不过缓冲区依然是满的，如果再有下一次接收操作，那么缓冲区就会留出空位，再次写入就不会阻塞了。

有缓冲channel缓冲区未满的接收部分
缓冲区未满的数据接收情况就比较容易理解了。如果存在接收变量，那就把数据项直接拷贝到接收变量里，然后清空缓冲区中对应的内存区，空出一个可用位置。然后递增recvx，移动到一个需要被读取的数据项。并递减qcount成员的数值，因为数据项已经被读取出去了。

阻塞部分
接收的阻塞部分和发送的阻塞部分差不多，不同的是接收操作的阻塞原因是：接收者在没有发送者阻塞或者没有发送者将数据项写入缓冲区的情况会阻塞。和发送操作阻塞一样，先是获取一个runtime.sudog，然后将当前goroutine打包放入接收阻塞链表并阻塞当前goroutine。等都有发送者的时候会被唤醒，然后校验等等。还有就是channel关闭之后唤醒的发送者不会panic，而关闭channel后唤醒的发送者会panic。

关闭

关闭一个未初始化的channel会引发panic，重复关闭也会panic。关闭操作会将所有当前阻塞在这个channel中的goroutine唤醒，无论是发送者还是接收者。其中，唤醒后的发送者将会panic，而接收者不会。
