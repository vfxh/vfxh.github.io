进程

进程：程序在操作系统中执行时的一个实例，管理操作系统分派资源的基本单位。进程的两个基本要素是程序代码（要执行的指令）和相关数据集（进程局部变量，全局变量，申请的内存空间等），其中程序代码可以被多个进程共享（当进程派生出子进程的时候，该进程和该进程的子进程使用的是同一份程序代码）。进程有几个重要的特征，分别是：动态，并发，独立，异步

特征 说明
动态 程序存储在某种介质上的指令集合，是静态的。而进程则是程序在操作系统中执行的一个实例，它是创建而产生，调度而执行，撤销而消亡。一个进程从创建到消亡之后就不复存在了，所以，进程具有动态性
并发 在内存中通常存在多个进程，这些进程在一段时间内可以同时执行，这种现象叫并发
独立 进程之间是相互独立存在的，拥有操作系统分配给它们的资源，运行也是独立的
异步 各个进程之间没有特别控制的情况下按各自独立的，不可预知的速度执行。它们互不干扰，互不影响

进程控制块PCB
进程控制块PCB（process control block）指的包含一些和进程相关的数据结构，在进程切换状态的时候需要读写这个进程控制块。比如暂停一个进程的时候，为了防止数据丢失，需要将相关的数据写入进程控制块。当恢复一个暂停的进程继续执行的时候，需要读取这个进程控制块来将相关的信息恢复到暂停前的状态。

进程控制块的一些主要成员

名称 说明
进程ID 进程的唯一标识符，用于唯一标记一个进程，来区别于其它进程
状态 进程的状态，比如执行时为运行态，暂停时为阻塞态
程序计数器 包含下一个执行的指令地址
优先级 相对于其它进程的执行顺序，如果一个进程优先级高，那么这个进程有限执行
上下文 进程执行时处理器寄存器中的数据，比如栈指针，临时存放在寄存器等一些数据
I/O状态信息 包括I/O请求，分配给进程的I/O设备和被进程使用的文件列表等
记账信息 比如进程使用处理器时间总和，进程占用处理器时间片等信息
… …

进程状态

从最直观的角度来看，一个进程在操作系统中要么处于运行状态，要么处于非运行状态。比如当一个进程被创建但是还没有运行就处于非运行状态，这个时候需要有一个存放非运行状态进程的容器，假设是一个队列。那么这个刚被创建的非运行状态进程就被放进这个队列里面。当正在运行的进程时间片到或者在等待一个事件发生，它就从运行态转变成非运行态，也被放进那个队列里面。这个时候处理器处于空闲状态，这就需要将某个处于非运行状态的进程调度到处理器执行。根据这个描述，就得到一个最简单的两状态模型。

两状态模型示意图

两状态模型对于直接观察到的进程状态来说是很直观的，但是对于考虑到多种情况的时候是不合适的。比如：非运行状态当中，有刚刚被创建的进程，也有因为某种原因而从运行状态转为非运行状态的进程。假如按照队列中先进先出的思想，一个进程是由于等待事件的发生而转为非运行状态。那么当过一段事件后，重新轮到这个进程继续运行。但是它等待的事件还没有发生，根本没办法继续执行。所以，区分不同的非运行状态的进程是很有必有的。根据这个问题，可以把非运行状态的进程分为 就绪态和 阻塞态。分别对应刚刚创建的进程和因为某些原因而转为非运行状态的进程。

创建和终止

现在考虑进程的创建和终止的问题。一个进程在被创建的时候，操作系统需要为这个进程分配资源。比如分配地址空间，用于管理这个进程的数据结构，分配唯一标识符等等。那么处于操作系统为其分配资源的进程是什么状态？如果将处于这个状态的进程也划分为就绪状态，假如这个进程被调度到处理器执行的时候还没有完成资源分配，那不出问题了吗？所以，这不能划分为就绪状态，而是将处于这个过程的进程的状态称为 新建态。触发进程创建的原因有很多，也许是有新的任务被提交，或者是由现有进程派生。比如一个TCP服务进程，为了能够在有限的时间内处理更多的请求。这个服务进程需要为每个请求提供一个新的进程去处理。这个时候，服务进程就可以显示请求操作系统去创建一个新的进程，这个动作就叫 进程派生。而创建的新进程叫做 子进程，请求操作系统创建进程的叫做 父进程 。

无论什么时候，进程的创建都是由操作系统进行的

进程的终止原因也有很多，可能是正常执行完成之后退出，这个是大多数进程的终止方式。也可能由于进程执行的指令无效引发的致命错误，或者操作系统资源紧张而被暴力终止。还有一种可能，就是父进程显示终止一个子进程，或者由于父进程终止而导致子进程的终止。而在进程终止的时候是直接退出什么都管，还是需要收集一些信息呢？很明显，当一个进程终止的时候，在完全消失之前操作系统需要收集一些信息。所以，进程终止分为两个步骤：一是让出处理器，把并且不在执行。然后操作系统收集相关信息，比如出错了之后的出错信息，进程执行过程中的记账信息，分析性能等等。二是收集完这些数据之后，删除与这个进程的相关数据。比如释放进程控制块，删除进程表中有关这个进程的进程表项等等。所以，进程终止的时候也需要有一个状态，退出态。当进程处于这个状态下的时候就表明操作系统正在做收尾工作。

五种进程模型说明

状态 说明
新建态 相应的进程控制块已经被创建，但是还没有被加载到内存中。但是前期的一些准备工作还没有完成。比如将程序代码从外存（虚拟内存）换入内存，在操作系统中的进程表为这个进程初始化对应的进程表项，给这个进程分配地址空间等资源
就绪态 进程的前期准备工作已经完成了，等待被调度去执行
运行态 进程正在被执行
阻塞态 可能是由于等待某个时间的发生而停止执行，比如等待I/O操作完成
退出态 进程可能是执行完任务了，或者由于致命原因被强制终止

Linux演示一个进程进行I/O操作的时候是否处于阻塞状态（文件I/O，网络I/O）

状态转换说明

状态转换 说明
空——>新建态 创建一个新的进程，然后这个进程处于前期准备时期
新建态——>就绪态 进程前期准备工作完成之后转成就绪态，等待被调度
就绪态——>运行态 在合适的时机被调度，然后执行
运行态——>退出态 进程执行任何完毕或者由于别的原因被强制终止，转为退出态，等待操作系统做收尾工作
运行态——>就绪态 正在运行的进程由于时间片用完，从运行态退出来。也可能是由于被其它优先级高的进程抢占
运行态——>阻塞态 进程在执行期间由于某些原因而暂停运行，可能是由于等待I/O操作完成。比如：在网络编程中，一个角色为服务端的网络应用在等待客户端的请求期间，就属于阻塞状态
阻塞态——>就绪态 进程由于等待某个事件发生而恢复到准备执行
就绪态——>退出态 某些操作系统父进程可以强制终止子进程（Linux）
阻塞态——>退出态 和上一条的说明相同

对状态转换的补充说明
时间片：操作系统为一个进程分配的“允许不中断”运行的最长事件段
进程间的抢占：操作系统会给不同的进程分配不同的优先级，优先级高的进程优先被调度去执行。假设有两个进程A和B，A的优先级比B的优先级高，并且只有一个处理器。A在运行期间由于等待一个事件而转成阻塞态，这时候B被调度变成运行状态。那么在B的运行期间并且时间片还没有用完，如果A等待的事件发生，A就从阻塞态转换成就绪态。由于A的优先级比B的优先级高，这个时候B被迫中断执行转成就绪态，A获取处理器。这种情况被称为抢占。

对于五种状态模型，如果只用一个队列来放置进程的话明显是不合理的。现在根据进程的五种状态模型来考虑一个问题，进程应该怎么放置。首先是新建态和退出态，处于新建态的进程被放置在虚拟内存中，所以不在考虑范围之内。退出态，可以只维护一个队列，然后转换成退出态的进程都放到这个队列里面，等到操作系统对其做的收尾工作完成之后删除掉。运行态的进程已经在执行，所以运行态的进程不在任何队列里面。现在就剩下阻塞态和就绪态，这两个状态涉及到优先级和事件的问题。假如对这两个状态只维护一个放置阻塞状态进程的队列和一个放置就绪状态的队列，可能会造成这样的局面：内存中有很多个分别处于阻塞态和就绪态的进程，它们的优先级以及等待的事件都不相同。当某个事件发生的时候，需要扫描整个阻塞状态的队列找到对应的进程。同样的，如果就绪队列中有优先级比运行态的进程要高的就绪进程，那么也需要扫描整个就绪队列，找到优先级高的进程优先调度，抢占正在运行的进程。因为在内存中的进程很多，这样只扫描单个队列效率是非常低的。
考虑这样一个方案，就是每个事件维护一个队列，有n个事件就有n个队列。每个优先级也维护一个队列，多个优先级有多个队列。当事件发生的时候，从某个事件的队列取出进程放到对应优先级就绪队列中，这样就高效很多了。

多个队列示意图

对上图的一些解释：

  1. 就绪态的进程在准备好了之后，被调入内存。操作系统会在初始化这个进程的时候给它一个优先级，然后根据优先级来放置到对应的就绪队列中
  2. CPU调度的时候，会根据进程的优先级和闲置的时间长短来决定调度哪个进程
  3. 由于等待某个而转为阻塞状态的进程，会根据等待的时间被放置到对应的阻塞队列中。比如等待I/O事件的进程会被放置到同一个阻塞队列中。
  4. 从阻塞队列唤醒的进程会根据其优先级被放置到对应的就绪队列中
  5. 当正在运行的进程被抢占之后或者时间片耗尽，会从运行态转为就绪态，根据其优先级被放置到对应优先级的就绪队列中

挂起

在五状态模型中，是假设处于阻塞态，就绪态，运行态等状态的进程都处于内存中。现在考虑这样的现象，如果在内存中的进程过多，导致内存不够用了怎么办。或者内存中的进程都处于阻塞态了，没有就绪态也没有运行态的进程。又由于内存已经达到最大的使用限度，新建态的进程无法调入内存中，阻塞态的进程等待的事件也不是立即发生。那么这个时间段的处理器就处于空闲状态，浪费资源。最简单的一个解决办法是增加内存，但是随着内存的增加，驻留在内存中的进程也会越来越多，还是会导致内存不够用的情况。另一个解决的办法是交换，交换：将进程的一部分或者全部调出内存，放到虚拟内存中的过程。由于交换被换出到虚拟内存的进程状态称为 挂起态。操作系统在虚拟内存中可以根据进程的优先级和等待事件等因素维护若干个挂起态队列，被换出的进程放置到对应的挂起队列当中。这样，换出一部分进程之后空余出来的内存空间，就可以用于存放被调入内存进程使用。被调入内存的进程可以是新建态的进程，也可以是之前被换出的进程。

进程可能一部分在内存中，一部分在虚拟内存中，等到在运行的过程中遇到不在内存中的部分才会调入虚拟内存的部分。但是，操作系统也会为了避免虚拟内存系统崩溃而把进程全部换出内存。

进程描述

前面说过，进程是管理操作系统分配资源的基本单位。那么操作系统就是资源的实体，它管理进程，为进程提供资源分配了调度等服务。操作系统为了管理进程和掌握分配给进程的资源状态，一般会维护四个表用来对进程以及分配给进程资源的管理。这四个表分别是：进程表，设备表，文件表，内存表，这里主要讨论进程表。进程表中有很
多个表项，每个表项保存着和进程有关的信息（可能进程表项中包含一个指向保存进程有关信息的结构的指针）。

示意图

在给进程下定义的时候有提到进程主要的两个因素：程序代码和相关数据。程序代码就是包含处理器要执行的指令集合。而相关数据是指，进程在执行的过程中涉及到的局部变量，全局变量，静态变量以及常量等数据。这些相关数据需要有特定的内存区来保存。在进程执行的过程中还会涉及到过程调用（函数调用）和参数传递，这个是通过一个称为栈的数据结构来保存的。除此之外，进程还包含很多属性，比如唯一标记某个进程的进程标识符，进程优先级，进程状态等等信息。这些属性集合保存在进程控制块中。而这些相关数据（程序代码，数据集）和栈，还有进程控制块的组合体称为 进程映像。操作系统维护的进程表中的每一个表项所包含的就是这个进程映像。

进程映像示意图（内存可能不是连续的）

进程在任意时刻，至少有一部分在内存中，或者全部在内存中。比如新建态的进程，它的进程控制块已经在内存中了，只要操作系统为这个进程做好前期准备完毕。就可以将部分或者全部进程映像调入内存。进程映像调入内存这个操作不是迁移而是复制，虚拟内存中保留的是进程映像的副本。所以，当内存中的部分进程映像被修改的时候（比如修改相关数据），还没有更新到虚拟内存中的进程映像前，虚拟内存中的进程映像就是过时的。

进程映像中进程控制块区域包含三个部分，现在对这三部分进行简单的说明：
进程标识信息： 包含进程标识符，进程所属的用户标识符，以及派生出该进程的父进程标识符等信息
处理器状态信息： 处理器上下文的部分信息，比如指向用户栈栈顶的栈指针，下一条要执行指令的地址等信息
进程控制信息： 比如进程的状态，进程的优先级，导致进程阻塞的事件，还有进程使用的资源信息（比如打开文件后的文件描述符，使用的内存地址等等）等

那么这个进程控制块是干什么用的呢？其实进程控制块除了包含上面提到的进程标识信息，处理器状态信息以及进程控制信息之外，还包含着指向其它进程控制块的指针。将多个具有相同状态的进程控制块串联起来形成一条链表。注意，串联起来的只是进程控制块本身。因为进程控制块本身还包含着和这个进程相关的资源，比如指向进程使用的内存块的指针，指向文件表项的指针等等。只要读取到进程控制块，就可以获取得到与进程相关联的信息。所以，进程控制块的作用其实是：操作系统用于管理进程，比如访问进程的相关信息，给进程分配或者回收资源等。总的来说，就是用来控制进程。之前讨论的状态模型中进程是存储在队列中的形式可以换成链表的形式，就像下图所示一样。

进程控制

执行模式
为了保护操作系统相关的重要资源，比如进程控制块等不受用于程序的干扰，一般情况下处理器至少支持两种执行模式。一种是与操作系统相关联的特权模式，也叫内核模式。另一种是与用户程序相关联的用户模式。当用户程序需要进行一些特殊的操作，比如父进程创建子进程或者强制终止子进程，I/O操作，中断后需要调用中断处理例程等等都需要从用户模式切换到内核模式。进入内核模式之前，处理器控制权需要从用户程序交给内核，等到从内核模式返回之后才会考虑是不是需要将处理器的控制权交还给之前的用户程序。

进程切换
引起进程切换的原因：

  1. 可能是当前进程的时间片已经用完了，然后操作系统会将该进程从运行态切换到就绪态，将另一个进程从就绪态切换到运行态，再把处理器控制权交给这个进程。
  2. 由于进程需要进行一次I/O操作，发生I/O中断。I/O操作并不会立即完成，因此等到I/O操作完成的进程会从运行态切换到就绪态，操作系统会在当前进行阻塞之后调度另一个就绪态的进程切换到运行态，并执行。
  3. 致命错误。当进程遇到致命错误的时候就会从运行态切换到退出态，当操作系统结束进程的收尾工作的时候就会撤销进程相应的资源，比如释放进程控制块等等。致命错误可能是由于进程占用的内存，过多，系统为了保证能够正常运行，不得已干掉这个进程。

进程切换和模式切换
模式切换可能不会导致进程切换。比如一个进程需要使用系统服务的时候产生了中断，当完成了系统调用之后，又重新将处理器控制权交换到之前的进程，这就不会发生进程切换。不发生进程切换的开销很小，只需要在进入内核模式之前保存相关上下文信息（相关寄存器数据，栈指针等栈信息，还有程序计数器等等）到进程控制块。从内核模式返回到用户模式之后，读取进程控制块恢复执行即可。

进程切换流程
进程切换需要做的工作很多，开销很大。进程从运行态切换到其它状态的大致流程：

  1. 保存当前进程的处理器上下文到进程控制块。处理器上下文包括：寄存器信息，栈指针，程序计数器等等。
  2. 更新当前进程的进程控制块，比如更新进程状态等等。
  3. 将当前进程放到相应的队列当中，比如阻塞队列，或者就绪队列。
  4. 选择一个合适的就绪态进程作为即将要执行的进程，并从就绪队列种取出。
  5. 更新合适的进程的进程控制块，比如就绪态更新为运行态。
  6. 从合适的进程的进程控制块中加载上一次别切换出运行态的信息，恢复运行现场，如果有的话。

线程

简单来说线程就是进程的其中一个执行轨迹，线程属于进程的一部分，一个进程有多个线程。进程拥有操作系统分配给它的资源所有权，进程内的全部线程共享这个进程的资源。在操作系统层面，进程是被调度和执行的实体。
