字节序

所谓字节序就是超过一个字节的数据类型的变量在内存中存储时字节的顺序，也就是说像char这种数据类型的变量不用担心字节序的问题了，因为char类型只占一个字节。字节序是处理器架构的一个特性，不同的处理器架构之间可能存在差异。所以，在同一台机器上的进程通信不需要考虑字节序的问题。而在不同主机之间进行通信，就需要处理字节序的问题了。字节序分为大端字节序和小端字节序，在TCP/IP协议体系中，进程间进行网络通信默认的字节是大端字节序。

大端字节序：最高有效字节出现在低地址上，最低有效字节出现在高地址上
小端字节序：最高有效字节出现在高地址上，最低有效字节出现在低地址上

比如现在有一个十六进制的整数0x12345678，有四个字节的内存块，这个数可以以这样的形式存储：0x12, 0x34, 0x56, 0x78。其中0x12就是最高有效字节，所谓的最高有效字节就是 权值最大的字节。相反的，最低有效字节就是 权值最小的字节，所以这里的0x78就是最低有效字节。

主机字节序和网络字节序
其实大端字节序和小端字节序只是计算机本身的说法，所以也叫主机字节序，只不过不同CPU架构的主机的字节序有可能不同而已。但是在网络传输中必须要统一字节序，否则某一端接收到其它端传输过来的数据就会发生混乱。所以，网络中使用的字节序需要相同，这就是网络字节序。比如TCP/IP协议体系使用的大端字节序作为网络字节序。

运输层

几个注意点：

  1. 运输层最大报文长度
  2. 网络层最大报文长度
  3. 网络层的IP协议分段可能会导致报文过大的UDP数据包数据被切割，TCP报文段重组之后不连续

运输层与网络层

运输层是介于应用层和网络之间，提供在不同主机之间的运行的进程进行逻辑通信。所以的逻辑通信是指，两台不同主机上的进程好像直接相互通信一样的方式。实际上两台主机之间的通信需要通过多种不同类型的链路相连接，经过多个路由器才能够彼此通信。而运输层提供的逻辑通信可以让不同主机上的进程可以不用考虑这些承载报文的基础设施细节，而直接相互传输报文。所以，运输层协议是是现在端系统，而不是在路由器上的。

运输层逻辑通信示意图1

实际上A和B两个应用进程的通信并非这么简单。现在假设A是发送端，B是接收端。A在接收到应用层进程传下来的报文后，需要转成运输层的报文。实现的方式根据使用不同的运输层协议而不同。如果是UDP协议，那么不会应用层传下来的协议做其它操作，再加上必须的UDP报文首部之后直接传给网络层。如果是TCP协议，则是对应用层传下来的报文转成字节流，然后根据情况选择若干个字节流作为传输数据再加上TCP首部传给网络层。网络层再将运输层传下来的报文组成网络分组，加上相应协议的报文首部之后向目的地发送（这里还涉及到链路层和物理层，不过这里就不讨论了）。在经过若干个网络路由器之后，B端接收到这个报文。B端的网络层剥离该报文的网络层协议首部，提交给运输层。运输层接收到报文之后对报文进行处理，做校验等操作。然后剥离运输层相应协议报文首部，将数据提交到应用层使用。还有一点需要注意，网络路由器仅作用于该数据报的网络层字段，对其中运输层的数据不会检查。

应用层和运输层会受制于低级别协议层，比如网络层会为上层协议提供时延带宽。如果网络层的时延很大，或者带宽不够，那将会直接影响到上层报文在传输过程中的效率

示意图2

从示意图2来看，运输层和应用层是工作在两个端系统中。而网络路由器只包含网络层，链路层和物理层。也就是说，网络层并不会处理也不会识别运输层和网络层的任何数据。

运输层概述

位于运输层之下的网络层为运输层提供服务，网络层有个叫做网际协议IP（Internet protocol）的网络协议为运输层中协议提供主机与主机之间的逻辑通信。每一台网络中的主机都有一个唯一标识这台主机的标识符，称为IP地址，IP地址的作用是为了定位网络中的主机。另外，IP协议提供一种不可靠的，尽最大努力交付报文的服务。也就说，如果没有另外保证可靠报文传输的方式干预，报文有可能会丢失在网络中。而且，IP协议提供的服务也不会保证报文的完整性，以及报文段（将一个报文拆分成一个个独立的段）的交付顺序。因此，如果想要达到报文在传输到目的主机的时候保持报文的正确性，就需要上层协议加以控制。比如运输层中的TCP协议。
运输层上中两个主要的协议，分别是TCP（transmission control protocol）和UDP（user datagram protocol）。其中，TCP协议的报文称为TCP报文段，UDP报文称为UDP数据报。TCP和UDP都会对应用层交付下来的报文或者网络层交付上来的报文提供差错检测。不过不同的是：TCP为应用层提供一种可靠传输的，按序交付的服务。也就是说，应用层使用TCP协议作为运输层支撑协议进行通信的过程中，在网络不断开连接的情况下，报文会完整的交付到对方的进程中。但是UDP协议则不然，它和IP协议同样是不可靠传输的，不保证报文完整性的协议的。有可能发送的报文在网络中丢失，也有可能到达对方主机上的报文不完整。不过，不论是TCP协议还是UDP协议，在通信的过程中，它们都是用同一个网络层协议，就是IP协议。
还有一个需要注意的是，既然网络层提供主机与主机之间的逻辑通信使用主机的IP来定位网络中一台主机的，那么运输层提供的进程与进程之间的逻辑通信受怎么定位哪个进程呢？换句话说就是，怎么知道报文要交付给哪一个进程呢？答案是 套接字。套接字是运输层和应用层进程之间进行数据交互的接口，套接字由ip和端口号组成。而在每一个需要网络服务的进程，都会有多个套接字。由于套接字是端口组成的，而一台主机有多个不同的端口号，所以一个端口号唯一标识一个套接字。简单来说就是，端口用来定位在一台主机上的某一个进程。所以，IP地址是为了定位主机，端口号是为了定位进程。端口号只有在同一台主机上才有意义，也就是 端口号只具有本地意义。不同主机上相同的端口号是没有关联的。
有些端口号是被限制的，用来分配给某些特定协议的。比如80端口被保留给HTTP协议使用，21端口是FTP协议使用，像这类端口称为熟知端口。目前端口号在UDP或者TCP协议中占两个字节，也就是65535，包括0在内的话就是65536个端口号。除了一些被限制的端口外，其它端口是可以自定义的。比如在实现网络应用服务端的时候可以指定应用程序监听哪个端口号（同一台主机一个端口号同一时间只能被一个进程监听。当然，一个进程可监听多个端口号，也就是说一个进程可以有多个套接字），而对于客户端来说，只会在它发起网络请求的时候动态分配，是一种短暂的端口号。
端口号在运输层协议报文中占两个字节，一台主机最大可分配65535个端口，0号端口一般不使用。

TCP中的报文按序交付的服务是指最终进程接收到的报文数据是从发送端发送的报文数据一样的顺序，但是在通信的过程中，TCP报文段的顺序可能不是按序到达的。具体参考TCP协议部分。

复用和分用

上文提到，UDP协议和TCP协议在通信的过程中都会使用同一个网络层协议，IP协议。在发送端将报文发送出去的时候，如果层运输层开始考虑，那么无论是UDP数据包还是TCP报文段在交付到网络层的时候，都需要经过IP协议处理。像这样同一层的多个协议实体都可以使用直接下层的同一个协议来传输数据的情况叫 复用，或者多路复用。而在报文到达通信的另一方接收端的时候，网络层的IP协议也可以将剥离出来的运输层报文交付到UDP协议实体或者TCP协议主体。像这样某一层的某个协议直接服务于上层的多个协议实体的情况叫 分用，或者多路分用。

运输层和网络层之间的复用和分用示意图

复用和分用不仅仅是在运输层和网络层之间才会出现，运输层和应用层之间同样会有。比如现在有两个使用应用层协议网络应用A和B，A使用的协议是HTTP，B使用的协议是FTP。这两个应用会将报文交付到运输层对应的协议实体，由于HTTP协议和FTP协议都是基于TCP协议。所以，这个交付的过程会涉及到复用。相应的，报文到达目的主机的时候，运输层将报文交付到应用层涉及到的情况就是分用了。分用是根据端口号的不同来决定将报文通过对应的套接字交付给对应的进程。

UDP协议

UDP协议只提供对直接上层应用层和直接下层网络层之间复用和分用功能，以及报文的校验功能。除此之外没有提供其它功能，即不提供可靠传输服务。对应用层交付下来的应用层报文加上必要的数据，比如源端口号和目的端口号，还有校验数据之后直接交付给网络层。当UDP数据到达目的主机之后，会根据报文中的目的端口交付给对应的进程。不过，虽然UDP协议本身不提供可靠传输服务，但是可以在应用层的程序实现可靠传输服务。比如增加确认和重传机制等，不过，这会导致大部分工作都会倾斜到保证UDP可靠传输的工作上，得不偿失。
UDP协议的特性：

  1. 无连接，两台不同主机之间的进程使用UDP协议通信前不需要建立连接
  2. 面向报文，即：UDP协议不管应用层交付下来的报文有多大，或者在运输层运行的进程使用UDP协议工作所发送的报文有多大，它都在加上必要的UDP首部之后直接交付给网络层。由于网络层可能会对报文进行分段，而导致最终传到对方主机上的UDP数据报不是完整的报文
  3. 尽最大努力交付，报文在网络传输的过程中可能会丢失
  4. 支持一对一，一对多，多对一，多对多的通信方式
  5. 首部开销小，只有8个字节

由于UDP的特性，使用UDP协议的服务相对来说效率高一点。因为不需要在通信之前消耗掉建立连接的时间，而发送速率不会因为网络阻塞这样的问题而降低。相应的，由于这些原因，发送的UDP数据报最终也有可能无法到达对方的主机。

从UDP报文的全部字段中，除了端口号和长度以及校验和之外，没有任何有关IP地址的字段。目的端口号是为了定位目的进程的，那么怎么定位到目的主机呢？答案是网络层的IP协议首部。之前有提到过，端口是为了定位目的进程，而IP则是为了定位目的主机。IP首部中包含目的主机的IP地址，所以，即使UDP报文中没有目的主机的IP地址，也能依靠IP数据包中的目的主机IP来找到接收端主机。

可靠传输原理

可靠传输是指：端和端之间进行通信的时候，数据通过一条可靠的信道进行传输。数据在传输的过程中不会发生损坏或者丢失的情况，并且多个数据在同一信道上从一端发送到另一端在交付的时候是按照发送顺序来交付的。

这里顺便提一下，TCP报文段交付到接收端是按照发送顺序的。但是TCP协议本身是面对字节流的，也就是说：发送TCP报文段中的字节到达对方主机可能不是按顺序到达的。但最终发送的字节组成报文段的时候，是按照发送报文段的顺序交付的。

如果某一个协议实现可靠传输机制，那么即使它对应下层的协议不提供可靠传输，这个协议也可以在网络上进行可靠通信（比如TCP实现了可靠传输，下层的IP协议是不可靠的）。而其对应上层协议即使也没有实现可靠传输，也可以依赖于该协议的可靠传输机制来在网络中进行可靠通信（比如HTTP协议依赖于TCP协议的可靠传输机制）。所以，如果处于应用层的协议基于运输层的UDP协议来进行网络通信，即使UDP协议不提供可靠传输机制，也可以在应用层实现可靠传输机制来进行可靠通信。不过这样子就很复杂，可能精力都放在实现可靠传输机制上了，脱离业务，得不偿失。

完全可靠信道传输

假设有一个完全可靠的信道，在这个信道传输的报文不会丢失，也不会乱序。也就是说，在这个完全可靠的信道上进行通信不会出现任何差错。在这种情况下，发送端收到由上层实体交付下来的报文就可以直接发送出去，然后回到等待上层实体交付报文的状态。接收端在完全可靠信道上接收一个报文，将这个报文交付给上层实体，然后回到等待从完全可靠信道上继续接收报文的状态。这样接收端和发送端就各有一个事件和一个动作，事件的发生会引起动作的执行。对于发送端，它的事件是接收上层实体交付它的报文，然后触发一个将报文发送出去的动作。对于接收端，其事件是从底层完全可靠信道接收一个报文，然后触发一个向上层实体交付报文的动作。

TCP编程

socket在Linux中是一个结构体，表示一种特殊的I/O，在Linux系统中每一个本地的socket都一个唯一的文件描述符来唯一标记，由操作系统分配。可以根据这个描述符来读写数据，从文件描述符读取数据，或者向描述符写入数据。

创建socket，socket在内核中创建。成功返回文件描述符，出错返回-1。

domain参数
AF_INET IPv4因特网域
AF_INET6 IPv6因特网域
AF_UNIX Unix域
AF_UNSPEC 未指定

type参数
SOCK_STREAM：用于TCP协议，流式套接字，面向连接的通讯流，提供可靠的，按序到达的连接。
SOCK_DGRAM：用于UDP协议，数据通过相互独立的报文进行传输，无序，不保证可靠性。
SOCK_RAW：原始套接字，允许对底层协议进行访问，比如IP或者ICMP进行直接访问。主要用于测试新的网络协议。
SOCK_SEQPACKET：长度固定，有序，可靠的面向连接的报文传递。

protocol参数
通常为0，表示按指定的域和套接字类型选择默认协议
